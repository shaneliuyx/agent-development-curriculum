# Tree-Construction Algorithm Comparison — Research Report

## Executive Summary

**Verdict: No published 2024-2026 algorithm has demonstrated measurably better RAGAS-style scores than W2.7's PageIndex-style build approach on a comparable benchmark.** The closest competitors — RAPTOR (bottom-up clustering), HiChunk (fine-tuned LLM chunker), BookRAG (tree+entity-graph), FABLE (forest+bi-path) — either (a) measure a different thing (chunking F1, retrieval recall, multi-hop EM), (b) use a different corpus (M3DocVQA, GutenQA, HiCBench, DragonBall), or (c) outperform vector RAG by structural reasoning while *agreeing with* (not replacing) the top-down LLM-assembled-tree builder W2.7 already uses. W2.7's pattern (top-down LLM-assembled hierarchy + recursive split + fact-rich summaries) is the same pattern PageIndex Cloud / Mafin 2.5 used to hit 98.7% on FinanceBench. The realistic upgrades are *vision-native parsing* and *fine-tuned chunk-boundary detection* — neither is a different tree-construction algorithm.

## What "tree-build quality" means for RAG retrieval

Every tree-RAG system has two orthogonal axes: (1) **Build** — bottom-up clustering+summarize (RAPTOR) vs top-down structure detection (PageIndex/W2.7/HiChunk/BookRAG) vs layout-first segmentation. (2) **Traversal** — cosine-on-collapsed-tree (RAPTOR), LLM-guided best-first (PageIndex/W2.7), bi-path LLM+vector (FABLE), Auto-Merge (HiChunk), entity-foraging agent (BookRAG). In 2024-2026 published work, *almost all reported deltas come from the traversal axis*; pure build-axis ablations are rare and small.

## Findings

### Finding 1 — RAPTOR's build-time advantage in isolation is small

Sarthi et al.'s canonical build-only ablation (RAPTOR clustering+summarize vs recency-window+summarize, identical retrieval on QASPER) shows RAPTOR wins by low single digits of accuracy. The 20pp QuALITY SOTA came from coupling with **GPT-4 at answer-time**, not the clustering builder. RAPTOR doesn't refute top-down structure-preserving builders; it shows *some* hierarchy beats none.

### Finding 2 — PageIndex Cloud / Mafin 2.5 is the same algorithm family as W2.7

Mafin 2.5 / PageIndex Cloud pipeline: vision-native PDF parser → TOC detection → tree assembly → recursive splitting → node enrichment → JSON. Structurally identical to W2.7's `build_tree.py`. Differentiators are (1) PageIndex OCR (long-context vision LM preserving headings/tables across two-column layouts), (2) GPT-4o-class LLM, (3) Cloud OCR vs `pypdf`. The repo README states: "This package uses standard PDF parsing. For complex PDFs, our Cloud Service offers enhanced OCR." The 98.7% is an OCR+LLM-quality story on a build pattern W2.7 already replicates.

### Finding 3 — HiChunk: learned chunk-boundary detection beats heuristics on chunking-sensitive benchmarks

HiChunk (Tencent 2025-09) fine-tunes Qwen3-4B for chunk-point prediction. With Auto-Merge it lifts evidence recall 74.06 → 81.03 on HiCBench T1 (evidence-dense). On evidence-sparse benchmarks (GutenQA, OHRBench) the gap collapses to <1pp. Financial reports skew evidence-dense, so this plausibly helps W2.7 — but the metric is chunk-boundary F1 / evidence recall, not RAGAS judge-aggregate, and not on FinanceBench.

### Finding 4 — BookRAG: tree-alone is the baseline it beats, not its contribution

BookRAG includes "Tree-Traverse… inspired by PageIndex" as a baseline and beats it by ~31 EM on M3DocVQA. The headline is for *BookIndex = tree + entity graph + agent*; tree construction itself is layout-parsing-driven hierarchical tree (W2.7 family). The lift comes from the entity graph and Information-Foraging agent. Signal: **adding an entity layer on top of the tree** is the next research direction — an *index extension*, not a tree-build replacement.

### Finding 5 — FABLE confirms top-down LLM-built tree is the consensus base

FABLE (2026-01) explicitly tabulates "bottom-up tree (RAPTOR clustering)" vs "top-down tree (LLM-generated hierarchy)" and chooses the latter. W2.7/PageIndex top-down is the 2026 consensus. FABLE's wins (92% completeness on DragonBall, beats gemini-2.5-pro at 4K tokens) come from bi-path retrieval, not the builder.

### Finding 6 — No study decomposes build vs traversal on FinanceBench/RAGAS

Closest decomposition is OpenRag MultiHop-RAG: "+RAPTOR hierarchy" gives Recall@10 +7.3pp over flat semantic; reranking adds +9.3pp on top — **build is half as impactful as traversal/reranking**. No published ablation holds W2.7-style structural build fixed and swaps in RAPTOR clustering on FinanceBench/RAGAS. Real gap in the literature.

### Finding 7 — Independent PageIndex benchmark caveat

Alden Do Rosario (2026-01) on SimpleQA-Verified ~1000 docs: PageIndex can't use its tree multi-document (too slow per doc), falls back to FAISS, lands at 0.69. Validates: (1) PageIndex-style build is single-doc tech, exactly W2.7's regime; (2) the 98.7% FinanceBench number is honest for its setup.

## Comparability caveats

1. **Eval mismatch.** W2.7's 0.79 is custom 8-Q Berkshire LLM-judge. RAPTOR's 82.6 is QuALITY accuracy. PageIndex's 98.7 is FinanceBench. HiChunk's 81 is HiCBench T1 evidence recall. BookRAG's 43.8 is M3DocVQA EM. None are RAGAS-aggregate on a shared corpus; cross-benchmark "X beats Y" is unsound.
2. **N=8 is too small for RAGAS claims.** W2.7's 0.79 has wide CIs; gap to vector (0.25) / graph (0.48) is meaningful, but 0.79 vs 0.85 would not resolve.
3. **Build vs whole-system attribution rarely reported.** Most papers report whole-system gains.
4. **OCR/parsing dominates gaps for financial PDFs.** Mafin 2.5's edge over OS PageIndex is largely PageIndex Cloud OCR.

## Recommendation

**Do not swap W2.7's tree-builder.** Top-down LLM-assembled hierarchy + recursive split + fact-rich summary is the 2026 consensus base (PageIndex Cloud, FABLE, KohakuRAG, BookRAG-base). Realistic next investments by expected RAGAS lift:

1. **Better PDF parsing** (vision-native OCR replacing `pypdf`) — closes gap to PageIndex Cloud's 98.7%; parsing, not building.
2. **Entity-graph layer on top of tree** (BookRAG-style) — +18 EM on M3DocVQA; index extension, not replacement.
3. **Auto-Merge dynamic-granularity retrieval** (HiChunk-style) — traversal-side.
4. **Fine-tuned chunk-boundary detector** (HiChunk Qwen3-4B) — replaces regex heuristic; only if Step 1 leaves residual errors.

The build algorithm is not the bottleneck; OCR, traversal, and entity layering are.
