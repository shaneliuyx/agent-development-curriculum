# Proposal: W3.5.9 — Requirement-Driven Memory Architecture

**One-line summary.** Add a new chapter (W3.5.9) that teaches readers how to choose between memory architectures — using LongMemEval as the real-world requirement, comparing Mem0 (open-source) against the W3.5.8 2-tier baseline and a homebrew hybrid router. Trim W3.5.8 so it stops over-committing readers to detail before the architecture-choice exercise lands.

---

## Pedagogical motivation

W3.5.8 currently teaches **ONE memory architecture deeply** (2-tier: operational quest queue + semantic store, consolidated periodically). It implies the 2-tier shape is canonical for agent memory.

The empirical evidence (W3.5.8 §8.7's 20-Q LongMemEval slice + Mem0's published 94.4 on the full benchmark) shows the 2-tier shape is **one of several viable patterns** and **NOT the right default for chat-shaped memory**. Production systems (ChatGPT memory, Claude memory, Mem0, Cursor) overwhelmingly use 1-tier write-time-consolidated memory; 2-tier is appropriate for narrower scenarios (task-completion agents, multi-agent shared queues, narrative+profile use cases).

The chapter currently teaches the JUNIOR pattern by accident: "here's an architecture, let me show you it works." The SENIOR pattern is "analyze the data, derive requirements, pick an architecture, verify, generalize."

W3.5.9 teaches the senior pattern explicitly, with LongMemEval as the worked exercise.

### §1 Why This Week Matters (~150 words)

Memory architecture chapters typically teach ONE pattern. Production engineering REQUIRES choosing the right pattern for the workload. This chapter teaches the meta-skill: given a real benchmark (LongMemEval), analyze the data shape, derive memory requirements per axis, evaluate three candidate architectures (1-tier atomic-fact, 2-tier consolidation, graph-tier temporal), pick (possibly compose), implement, and verify. The deliverable a reader walks away with: a decision framework + a working hybrid implementation, not memorised facts about any single backend. Interview signal: "how do you decide?" is harder than "how do you implement?" — this chapter answers the former.

### §2 Theory Primer (~1000 words)

- Three architecture classes side-by-side (1-tier Mem0, 2-tier W3.5.8, graph Graphiti)
- Mem0's 94.4 on LongMemEval — full benchmark numbers + per-axis breakdown
- How question shape determines architecture choice
- Hybrid architectures: router patterns and when they're worth the operational overhead
- Decision matrix template

### §4 Lab Phases

#### Phase 1 — Requirement analysis from LongMemEval (~1 h)

Reuse `data/longmemeval_slice_w358.json` (from W3.5.8 §8.7) and extend `scripts/build_slice.py` to cover all 6 question_types (currently only multi-session + knowledge-update).

For each question type, decompose along 5 requirement dimensions:

| LongMemEval axis            | Atomic-fact recall | Episode narrative | Bitemporal | Profile aggregation | Cross-session |
|-----------------------------|--------------------|--------------------|------------|----------------------|----------------|
| single-session-user         | ✅                  | —                  | —          | —                    | —              |
| single-session-assistant    | ✅                  | —                  | —          | —                    | —              |
| single-session-preference   | —                   | —                  | —          | ✅                    | —              |
| multi-session               | ✅                  | ✅                  | —          | —                    | ✅              |
| knowledge-update            | ✅                  | —                  | ✅          | —                    | ✅              |
| temporal-reasoning          | ✅                  | —                  | ✅          | —                    | —              |
| abstention (_abs suffix)    | orthogonal — retrieval-gate concern, not store-class concern                            |

Output: requirement vector per question type, derived empirically from real sample inspection — not assumed.

#### Phase 2 — Architecture decision (~1 h)

Apply Phase 1 requirement vectors against the 3-class decision matrix:

- Information Extraction / Multi-session / Knowledge-update → **1-tier (atomic-fact) wins**
- Profile-aggregation single-session-preference → **2-tier with profile tier wins**
- Temporal-reasoning → **1-tier with timestamps OR graph-tier**
- No single class wins all axes → **hybrid router justified**

#### Phase 3 — Open-source baseline: Mem0 (~2 h)

- Install Mem0 (`pip install mem0ai`)
- New module `src/mem0_backend_adapter.py` (~80 LOC) — thin wrapper conforming to W3.5.8's `TieredMemory`-style interface
- Add `--backend mem0` flag to `src/run_longmemeval_slice.py` (~30 LOC dispatch)
- Reuse `scripts/aggregate_results.py` unchanged
- Re-run on the same slice with the same reader (gpt-oss-20b) + same judge (claude-sonnet-4-6)

#### Phase 4 — Homebrew hybrid router (~3 h)

- `src/atomic_fact_memory.py` (~120 LOC) — 1-tier path: per-message atomic-fact extraction (one LLM call per message, JSON array output) → Qdrant
- `src/router_memory.py` (~150 LOC) — question classifier (small LLM call OR rule-based) → dispatches to `atomic_fact_memory` (1-tier) or `tiered_memory_qdrant` (W3.5.8 2-tier)
- Same interface as existing backends; `run_longmemeval_slice.py` accepts `--backend hybrid` with no further refactoring

#### Phase 5 — Compare + aggregate (~1 h)

Add three new backend rows to results: `mem0`, `atomic_fact_only`, `hybrid_router`. 5-backend × 6-axis × 20-Q comparison table.

Expected outcomes (calibrated against our M5 Pro + gpt-oss-20b stack):

| Backend                              | Estimated score |
|--------------------------------------|------------------|
| Qdrant + summarize (W3.5.8 §8.7)     | 0/20 (measured)  |
| EverCore + full pipeline (§8.7)      | 0/20 (measured)  |

Production-numbers gap explained: GPT-4-class reader + their tuned multi-signal fusion. The chapter REPORTS this gap honestly rather than chasing it.
